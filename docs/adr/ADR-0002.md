# ADR-0002 — Immutable Domain Objects

**Status:** Accepted
**Date:** 2026

---

## Context

-e Blockchain objects must satisfy invariants established at creation time. A Transaction with a valid signature must remain valid. A Block with a correct Merkle root must not have its transaction list silently modified. Allowing unrestricted mutation after construction makes these invariants difficult to preserve and harder to reason about across concurrent or asynchronous execution paths.

---

## Decision

-e Core blockchain objects in abyss-core become immutable after construction.

- Creation happens through constructors that validate all invariants.
- Validation happens during construction; an invalid object cannot be created.
- State changes happen only through explicit transition functions that return new values or update a separate state container.
- Setters on core types are absent by design.

---

## Consequences

-e Advantages:
- Predictable behaviour — an object's meaning does not change after it is created
- Easier testing — no need to set up mutation sequences
- Stronger invariants — validation logic is concentrated in one place
- Easier debugging — the creation site is always the source of truth

Trade-offs:
- Some patterns (builder pattern, staged construction) require care
- Update operations must return new objects or route through explicit state transition functions
