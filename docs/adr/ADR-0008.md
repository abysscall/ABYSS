# ADR-0008 — Hybrid Proof-of-Stake with BFT Finality

**Status:** Accepted (target mainnet; devnet uses single-proposer PoA)
**Date:** 2026

---

## Context

ABYSS requires a consensus mechanism that satisfies four constraints simultaneously:

1. **Instant finality** — the DEX and wallet UX cannot tolerate probabilistic finality
   ("wait six confirmations"). A transaction must be either final or not.
2. **Permissionless participation** — any holder of staked AC must be eligible
   to propose and validate blocks. No foundation-controlled allow-list.
3. **Privacy compatibility** — the consensus mechanism must not require
   validators to see the plaintext content of shielded transactions.
4. **Auditable decentralisation** — the validator set and stake distribution
   must be verifiable on-chain, not taken on trust.

Evaluated alternatives:

| Mechanism | Finality | Permissionless | Why rejected |
|---|---|---|---|
| Proof of Work | Probabilistic | Yes | Energy cost; slow finality incompatible with DEX UX |
| Pure PoS (probabilistic) | Probabilistic | Yes | Same finality problem as PoW for our use case |
| Proof of Authority | Instant | No | Fixed validator set contradicts decentralisation pillar |
| Delegated PoS | Near-instant | Partial | Small elected set creates centralisation risk |

---

## Decision

ABYSS adopts **Hybrid Proof-of-Stake with BFT finality**, in the tradition
of Tendermint / HotStuff consensus engines.

**Mechanism:**
1. Validators stake AC to earn block proposal rights. Proposal selection is
   weighted by stake.
2. A proposed block is broadcast to the full validator set.
3. Validators vote in a BFT round. Once ≥ 2/3 of total staked AC has signed,
   the block is finalised immediately.
4. Finalised blocks are irreversible under the safety assumption that
   no more than 1/3 of stake is Byzantine or offline.

**Current devnet status:**
The devnet runs a single-proposer model (effectively PoA) for rapid iteration.
This is a development convenience, explicitly not the production design.
Multi-validator BFT is Phase 3 on the roadmap.

---

## Consequences

- Instant, deterministic finality enables the DEX and wallet UX described
  in the whitepaper.
- Stake-weighted permissionless entry preserves the decentralisation pillar.
- BFT signing records are publicly verifiable — validator participation
  is auditable, not asserted.
- The 1/3 Byzantine fault tolerance threshold is a standard and
  well-understood safety bound.

Trade-off: BFT consensus requires O(n²) message complexity as the validator
set grows. This is manageable at typical validator set sizes (tens to hundreds)
but would require sharding or committee selection at very large scale.
This is a known future scaling problem, not an immediate concern.
