# ADR-0004 — Atomic Block Application

**Status:** Accepted
**Date:** 2026

---

## Context

-e A block may contain multiple transactions. If transaction N of M fails during execution, the state updates from transactions 1 through N-1 must not be persisted. Partial state updates would produce a ledger inconsistency that cannot be resolved without a chain reorg. This is a correctness requirement, not a performance optimisation.

---

## Decision

-e Blocks are executed on a temporary copy of the chain state (or a write-ahead journal).

If execution succeeds for all transactions:
- The temporary state replaces the current state atomically.

If any transaction fails:
- All changes from the entire block are discarded.
- The chain state is identical to its state before the block arrived.

This is implemented in abyss-core's chain.produce_block path.

---

## Consequences

-e Guarantees:
- Deterministic execution: the same block always produces the same state transition
- Consistent state: partial updates never reach the committed ledger
- Easier recovery: a failed block leaves no trace
- Simpler testing: block application is a pure function of state and block
