# ADR-0016 — Genesis Allocation Registry

**Status:** Accepted
**Date:** 2026
**Depends on:** ADR-0015 (native-first launch), ADR-0012 (supply and sale structure)

---

## Context

ADR-0015 removes ERC-20 and any temporary token from the critical path. That
raises a concrete question: if no token exists before mainnet, how are
pre-genesis contributions recorded and turned into balances?

The answer must satisfy three properties:

- **No temporary token.** No ERC-20, no testnet coin, no escrow token stands in
  for AC. AC first exists at genesis and nowhere earlier.
- **Verifiable.** A contributor — and later an auditor — can prove exactly which
  allocation a contribution produced, and that the genesis balances match the
  recorded contributions.
- **Deterministic genesis.** The first mainnet balances are a pure function of
  the registry, so any node can independently rebuild and verify the genesis
  state.

## Decision

Introduce a dedicated **Genesis Allocation Registry** — its own module, later
folded into a Genesis Builder — with the following pipeline:

```
Contribution → Contribution Receipt → Genesis Allocation Registry
            → Genesis Root → Genesis Builder → first Mainnet AC balances
```

### 1. Contribution Receipt

A signed, immutable record created after a contribution payment is confirmed:

| Field                 | Meaning                                                  |
|-----------------------|----------------------------------------------------------|
| `contribution_id`     | Unique identifier for the contribution                   |
| `timestamp`           | When the contribution was confirmed                      |
| `amount`              | Amount paid, in the paid currency's smallest unit        |
| `currency`            | BTC / USDT / USDC / bank / Lightning (external rails)    |
| `future_abyss_address`| The address that will hold the AC allocation at genesis  |
| `ac_allocation`       | AC granted, derived from amount + the ADR-0012 stage price|
| `signature`           | Digital signature by the registry authority              |
| `hash`                | Content hash of the receipt (integrity + Merkle leaf)    |

The external currencies are **payment rails only** — they are not, and never
become, the ABYSS token.

### 2. Genesis Allocation Registry

An append-only, ordered set of receipts. Responsibilities:

- reject duplicate `contribution_id`s;
- aggregate multiple receipts for the same `future_abyss_address` into one
  balance;
- enforce the ADR-0012 supply caps and per-stage token caps;
- expose a **Genesis Root**: a Merkle root over receipt hashes (reusing
  `abyss_core::hashing::merkle_root`) that commits to the entire allocation set.

### 3. Genesis Builder

At genesis, the builder consumes the registry and produces the `GenesisConfig`
allocations (`Vec<(Address, Coin)>`) that seed the ledger. Because this is a pure
function of the registry, the resulting genesis state is reproducible and the
Genesis Root can be published for independent verification.

**The Genesis Block literally mints the first AC** from these allocations. There
is no migration event and no swap contract — allocations *become* balances.

## Consequences

- ABYSS launches with a clean, auditable path from contribution to native
  balance, with no intermediate token and no migration risk — a strong argument
  for investors and auditors.
- The Genesis Root is a single verifiable commitment to the entire pre-genesis
  raise.
- Custody of received external funds and KYC/legal handling live *outside* the
  registry; the registry records allocations, not custody. Accepting funds still
  requires the legal review noted in ADR-0015.
- Implementation lands as an `abyss-genesis` (or `abyss-core::genesis_registry`)
  module reusing existing `Coin`, `Address`, and `merkle_root` primitives.

## Notes

Staged implementation: (1) `ContributionReceipt` + registry with cap enforcement
and Genesis Root; (2) Genesis Builder producing `GenesisConfig`; (3) wiring into
the node's genesis path. The existing `abyss-tokenomics::ContributionReceipt`
(currently a quote model) is the natural starting point and will be extended to
the fields above.
