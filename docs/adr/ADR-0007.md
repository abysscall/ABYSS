# ADR-0007 — Architecture Before Features

**Status:** Accepted
**Date:** 2026

---

## Context

-e ABYSS is designed to be an ambitious system: private, decentralised, AI-native, and social. The scope creates pressure to add features quickly to demonstrate progress. That pressure, if not actively resisted, produces systems that are wide but shallow — many features, none of them done correctly.

---

## Decision

-e ABYSS prioritises architectural quality over feature quantity.

New functionality must not compromise:
- Simplicity: the system should be understandable in pieces
- Correctness: behaviour must be specified and tested before it is shipped
- Maintainability: code written today must be readable by someone unfamiliar with it
- Deterministic behaviour: the same input always produces the same output

Every subsystem must justify its existence.
When uncertain, choose the simpler architecture.
If a simpler architecture is not yet apparent, wait.

---

## Consequences

-e Some features are deferred. This is intentional and correct.
The alternative — many features on a weak foundation — produces a system that must eventually be rebuilt from scratch.
