# ADR-0009 — Hybrid Execution Layer: EVM-Compatible + ZK Circuit VM

**Status:** Accepted
**Date:** 2025

## Context

ABYSS needs a smart contract execution environment that satisfies two conflicting requirements:

1. **Ecosystem adoption** — developers need to write contracts in familiar languages
   with existing tooling (Hardhat, Foundry, Remix, Solidity, existing audit expertise).

2. **Native privacy** — the EVM model is transparent by design. Shielded computation
   cannot be expressed inside the EVM without fundamental changes.

Two pure approaches fail:

- **Custom VM only** (e.g. Aleo/Leo): maximum privacy control, but developers must
  learn a new language. Ecosystem growth is slow. Auditors are scarce.
- **EVM only**: rich ecosystem, but privacy must be retrofitted via mixers,
  which provides weak guarantees and has drawn regulatory action.

## Decision

ABYSS uses a **dual-track execution model**:

- **Public track**: EVM-compatible environment for contracts where transparency is
  acceptable (DEX logic, governance, token standards, NFTs). Existing Solidity
  contracts can be deployed with minimal changes.

- **Private track**: ZK circuit VM for shielded transfers and confidential computation.
  Used for: shielded AC transfers, private smart contract state, Agent policy enforcement.

**Cross-VM boundary**: narrow and auditable. EVM contracts can verify that a valid
ZK proof was provided for a private operation, without seeing the underlying data.

## Consequences

- EVM compatibility attracts existing developers immediately
- ZK circuit track provides genuine privacy, not retrofitted obscurity
- Cross-VM boundary limits attack surface
- Two execution environments to maintain and audit (accepted cost)
- ZK circuit implementation is deferred to Phase 4 (see ROADMAP.md)
